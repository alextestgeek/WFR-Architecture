# 14. Готовность ядра и матрица прорывов / узких мест

**Дата:** 3 апреля 2026  
**Связь:** [`13-project-status-snapshot.md`](13-project-status-snapshot.md) (срез «где мы»), [`12-wfr-llm-breakthrough-roadmap.md`](12-wfr-llm-breakthrough-roadmap.md) (этапы A–F), [`07-experiment-plan.md`](07-experiment-plan.md).

Этот документ отвечает на два запроса: **(1)** как зафиксировать, что **ядро способно работать** в смысле репозитория; **(2)** где искать **прорыв** для LM и какие **узкие места** уже измерены, а какие нет.

## 1. Подтверждение: ядро работает (регрессия Phase 0)

Имеется в виду пакет `wfr.core` + проверенные постановки **без** спора о LM.

| Проверка | Что доказывает | Артефакт / запуск |
|----------|----------------|-------------------|
| Smoke | Сборка сети, forward, отсутствие «мёртвого» стека в базовой конфигурации | `experiments/00-smoke-test/run_smoke_test.py` |
| Memory & complexity | Память на токен не взрывается с длиной контекста (в рамках теста) | `experiments/01-memory-complexity-test/run_memory_test.py` — отчёт [`09-memory-complexity-test-plan.md`](09-memory-complexity-test-plan.md) |
| Layer scaling | Глубина до 32 слоёв с активными уровнями после v2.1 homeostatic | `experiments/02-layer-scaling-test/run_layer_scaling_test.py` + `03-theory.md` §8 |
| Long-context stability | Внутренняя стабильность паттернов при росте \(T\) | `experiments/03-long-context-stability/run_stability_test.py` |
| Pattern formation | Различимость позиционных паттернов (теория §5) | `experiments/04-basic-pattern-formation/run_pattern_test.py` |

**Минимальный «хвост CI» перед тяжёлой LM-сессией:**  
`run_smoke_test.py` + (по времени) один короткий прогон `02` или импорт `WFRNetwork` из актуального `wfr/core.py` в тесте Exp 09.

**Граница ответственности:** Phase 0 **не** гарантирует competitive CE на WikiText; это проверяется отдельно в Exp 08–09.

## 2. Узкие места в связке **ядро + WFRLM + LM** (как в коде)

Приоритеты и кандидаты **новых вводных** (формулы: каузальный контент, readout от слоёв, медленное состояние): [`03-theory.md`](03-theory.md) **§11**.

Независимо от теории, текущая цепочка имеет явные ограничители:

| # | Узкое место | Механизм в коде | Симптом на char-LM |
|---|----------------|------------------|---------------------|
| U1 | **Канал к vocab** | `readout_feat_dim`, MLP readout, `readout_wave_kernel` | Большой зазор при малых D; sweep по D снимает зазор; `rw` слегка улучшает при фиксированном D |
| U2 | **Токен → фазы** | `token_phase_offset` (сдвиг в \(M\) фазах) — см. [`03-theory.md`](03-theory.md) §10 | Сложные морфологические зависимости могут требовать BPE или более широкого контент-представления |
| U3 | **Локальность по времени в поле** | до недавнего времени — только по позиции `t`; теперь `phase_causal_kernel`, `content_neighbor_mix` | На A100 при `readout=16`: phase conv ≈ паритет с TF; neighbor mix — слабый эффект в quick |
| U4 | **Сигнал для обучения** | Full \(L\) vs CE-only; RFP отдельно | Неправильный вывод «WFR не учится», если сравнивать разные objective / ёмкость |
| U5 | **Масштаб данных и шагов** | `--wiki-push`, длинные эпохи | Кривые могут не сошлись на малом бюджете |
| U6 | **Дифференциация по контексту** | Заявка O(1) памяти vs Transformer | Для LM-**прорыва** нужен протокол этапа **D**: \(T_1 \ll T_2\), те же params/order, замер **качества и RAM/latency** |

## 3. Матрица «гипотеза прорыва → как проверить → статус»

| ID | Гипотеза (прорыв / рычаг) | Фальсификация / узкое место | Эксперимент (минимум) | Метрика успеха | Статус (2026-04) |
|----|-----------------------------|-----------------------------|------------------------|----------------|-------------------|
| H1 | Узкий readout — главный зазор с TF | Если при росте D зазор не сужается | Exp 09 fair-parity sweep `readout_feat_dim` | Δ CE ↓ с D; при D=32 Δ≤0 | **Подтверждено** на A100 |
| H2 | MLP readout даёт выигрыш при том же core | Линейный head достаточен | Parity с `--readout-mlp-hidden H`, match-capacity | Δ лучше vs linear при том же классе params | **Частично** (A100 B1, [`runs/04_b1_mlp_matrix_20260403`](../experiments/09-lm-parity/outputs/remote_a100/runs/04_b1_mlp_matrix_20260403)): при **D=16** MLP **увеличивает** Δ (+0.047→+0.132); при **D=32** MLP **улучшает** перевес WFR (−0.063→**−0.102**). Итог: MLP не универсален при узком D на этом протоколе; при широком D — полезен. Повторить на втором бюджете/сидах по правилу §4 п.6. |
| H3 | Каузальная смесь фаз критична для LM | Без неё тот же CE | `--phase-causal-kernel K` vs контроль | Устойчивый Δ&lt;0 при readout 16 | **Не подтверждено** (≈ничья на B3) |
| H4 | Локальная смесь **волны** перед головой | Голова не видит соседей по амплитуде | `--readout-wave-kernel K` | Δ лучше контроля | **Слабо подтверждено** (+0.047→+0.034) |
| H5 | Длинный train закрывает остаточный зазор | Оптимизация / недообучение | Этап **C**: те же модели, больше ep / данных | val CE продолжает падать; WFR не отстаёт | **В очереди** |
| H6 | WFR выигрывает на **длинном** контексте при равном бюджете | Память TF растёт быстрее | Этап **D**: пары \(T\), match-capacity TF с окном | Лучше CE или меньше пик RAM при сопоставимом качестве | **Не начато** для полной пары \(T_1 \ll T_2\) на A100; локальные черновики `longcontext_pair_*.json` (CPU) — см. [`verify_longcontext_artifacts.py`](../experiments/09-lm-parity/verify_longcontext_artifacts.py) |
| H7 | RFP даёт **предсказуемый** прирост после parity | Шум в метрике | Этап **E**: фиксированный протокол + версия RFP | Правило «если C → ΔM&gt;0» с \(p\) | После стабилизации B/C |
| H8 | BPE снимает ограничение char-алфавита | Слишком большой vocab при тех же params | Этап **B2**: тот же WL, subword | CE vs char при честном сравнении | **Не начато** |

## 4. Рекомендуемый порядок работ (следующий цикл)

1. **Регрессия ядра:** прогон smoke (и при возможности скalen тест) после крупных правок `wfr/core.py`.
2. **B1:** **закрыт** (2026-04-03): матрица в [`runs/04_b1_mlp_matrix_20260403`](../experiments/09-lm-parity/outputs/remote_a100/runs/04_b1_mlp_matrix_20260403); см. строку **H2** в §3. Повтор для §6 теории — по желанию второй прогон.
3. **B3 (combo):** один прогон **`readout_wave_kernel=3` + `readout_mlp_hidden`** при фиксированном **D=32** (или лучшей строке из B1) — аддитивность с MLP.
4. **C:** `wiki-push` или увеличение `PARITY_EPOCHS` **только после** фиксации лучшей связки из п.2–3.
5. **D (старт):** спецификация пары контекстов \(T_1, T_2\) и matched TinyTransformer с окном `max_len ≥ T_2`; отчёт: CE + **peak GPU memory** + время эпохи.
6. **Теория:** патч в `03-theory.md` только когда по каждому ряду H2/H4 есть **≥2** согласующихся прогона или явный fail. Быстрая регрессия следствий §9 против JSON: [`experiments/09-lm-parity/verify_theory_calibration.py`](../experiments/09-lm-parity/verify_theory_calibration.py).

## 5. Где **не** ждать прорыва без новой постановки

- Сравнение с Transformer **без** match-capacity / без общего val seed — не отвечает на вопрос об архитектуре.
- Один quick-прогон на CPU — только отбор гипотез; итоговые выводы по §2–3 — **GPU + бюджет как в Exp 09**.
- Ожидание, что `phase_causal` **сам по себе** заменит расширение readout после данных B3 при `readout=16` — **не подкреплено**; рычаг U1 остаётся главнее.
