# 10. Phase 1 — Обучение и RFP (Resonant Field Plasticity)

**Версия:** 0.2  
**Дата:** 31 марта 2026  
**Статус:** В работе (открыт после закрытия Phase 0)

**Связь с теорией:** [`03-theory.md`](03-theory.md), раздел 6 (RFP, целевая функция \(L\), RC, EnergyCost).

---

## 1. Цель Phase 1

Перейти от **чистого forward** (Phase 0) к **обучаемой** WFR-сети: проверить, что градиенты проходят через WPE, резонансные слои и surrogate spike; зафиксировать минимальные правила обновления параметров; подготовить почву для полноценного RFP (двухрежимность, явные \(\Delta f\), \(\Delta\theta\), \(\Delta\alpha\)).

Phase 1 **не** ставит целью SOTA на языковых бенчмарках — это **Phase 2+**.

---

## 2. Объём (что входит в Phase 1)

| Слой работ | Содержание |
|------------|------------|
| **1. Sanity** | Один forward–backward, отсутствие NaN; опционально краткое снижение loss на toy-задаче ([Experiment 05](../experiments/05-rfp-training-sanity/README.md)). |
| **1b. Целевая функция v0** | Реализация скаляра \(L = \alpha L_{\text{task}} + \beta (1-\overline{\text{RC}}) + \gamma L_{\text{energy}}\) с явными коэффициентами (хотя бы на малых данных). |
| **2. RFP v0** | Один режим обучения (online **или** offline), явное правило шага для выбранных параметров (частоты, `decay`, веса интерференции, \(\alpha,\beta,\gamma\) WPE при необходимости). |
| **3. Стабильность** | Наблюдение за RC и спайками при обучении; отсутствие взрыва loss на 1–2 порядка длины контекста и глубины из Phase 0. |

Полная двухрежимность RFP, глобальное выравнивание поля и формализация всех \(\Delta\) из теории — **итеративно**, через эксперименты внутри Phase 1.

---

## 3. Критерии успеха Phase 1 (мастер-список)

Критерии **наращиваются** по мере выполнения подэтапов.

### 3.1 Обязательные (минимальный «закрытие» Phase 1)

1. **Градиенты:** через цепочку WPE → резонанс → surrogate spike градиенты **конечны** и **не нулевые** хотя бы у подмножества обучаемых параметров (на toy-задаче). Experiment 05 (enhanced) дополнительно логирует L2-норму градиента на первом шаге.
2. **Обучение v0:** на явно заданной toy-задаче (например next-token с малым словарём) loss **снижается** относительно стартовой точки при фиксированном сиде **или** стабильно ниже baseline «случайного угадывания» (\(\approx \ln V\) для CE). В режиме **enhanced** то же проверяется на **отдельном val** наборе батчей по полному \(L\) и по CE.
3. **Документация:** зафиксированы используемые \(\alpha,\beta,\gamma\), определение \(L_{\text{energy}}\) (например средняя доля спайков по слоям), способ усреднения RC.
4. **Воспроизводимость:** скрипт эксперимента, README с параметрами и артефакт `outputs/*.json`.

### 3.2 Желательные (усиливают доверие к RFP)

5. Стабильность обучения при увеличении числа фрактальных уровней / слоёв в пределах тех же порядков, что в Phase 0.
6. Явное сравнение шага «только TaskLoss» vs полное \(L\) (RC + energy).

### 3.3 Не являются критериями Phase 1

- Качество на реальном корпусе или стандартных LM-бенчмарках.
- Сравнение с Transformer / Mamba на тех же длинах (это отдельный план, Phase 2+).
- Доказательство суб-линейного времени по \(n\).

## 4. Оговорки и ограничения

1. **Теория RFP** в [`03-theory.md`](03-theory.md) — **перспективная гипотеза**; полная формализация \(\Delta f\), \(\Delta\theta\), \(\Delta\alpha\) **подстраивается** под результаты Phase 1, а не наоборот.
2. **Surrogate gradient** в Phase 0 по обучению **не верифицирован**; Phase 1 **первым** подтверждает или опровергает работоспособность backward в типичных настройках (см. [`08-phase-0-plan.md`](08-phase-0-plan.md), раздел 7).
3. **Homeostatic regulation** при `model.train()` в текущем `wfr_core` **не обновляет** пороги (логика завязана на `not self.training`); поведение при обучении с включённым homeostastic в другом режиме — **отдельное решение** для RFP v1.
4. **Пороги спайка** (`spike_threshold`) по умолчанию **без градиента** (`requires_grad=False`); обучение сначала идёт по частотам, весам интерференции, параметрам WPE и голове — согласуется с акцентом теории на фазах/частотах, но это **упрощение**.
5. **Toy next-token** с подачей `token_ids` в энкодер как «позиций» — **инженерный трюк** для проверки градиентов, а не заявление о лучшей схеме для реальной LLM.
